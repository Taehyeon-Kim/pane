### Skill Manifest Specification

**Manifest File Format:**
The `pane-skill.yaml` manifest follows a standardized structure defined in the Skill data model. All fields are required unless marked optional.

**Complete Manifest Structure:**

```yaml
id: "claude-tips" # Required: Unique identifier (String)
name: "Claude Code Tips" # Required: Human-readable display name (String)
description: "Browse a curated archive of Claude Code usage tips." # Required: 1-2 sentence explanation (String)
version: "0.1.0" # Required: Semantic version (String)

exec: "claude-tips" # Required: Executable name or path (String)
args: [] # Required: CLI arguments (Vec<String>, can be empty)

tags: # Required: Searchable tags (Vec<String>)
  - tips
  - claude
  - coding

estimated_time: "1–3 min" # Optional: Human-readable time estimate (Option<String>)

ui:
  mode: tui # Required: "tui" or "inline" (UiMode enum)
  fullscreen: true # Optional: Whether skill uses fullscreen (bool, default: true)

context:
  pass_cwd: false # Optional: Pass current working directory (bool, default: true)
  pass_git_root: false # Optional: Detect and pass git root (bool, default: true)
  pass_project_name: false # Optional: Pass project name (bool, default: true)
  pass_stdin_json: false # Optional: Send full context as JSON to stdin (bool, default: false)
```

[Source: docs/architecture/data-models.md#skill]

**Field Explanations:**

- **id**: Unique identifier used for skill discovery, favorites, recent tracking, and ID collision resolution
- **name**: Display name shown in Pane launcher skill list
- **description**: Shown in detail pane, explains what the skill does
- **version**: Semantic version for skill versioning and updates
- **exec**: Executable name (if in PATH) or absolute path to skill binary
- **args**: Command-line arguments passed to executable on launch (empty for claude-tips)
- **tags**: Searchable keywords for fuzzy matching and filtering
- **estimated_time**: Human-readable time estimate shown in detail pane
- **ui.mode**: "tui" means skill takes over terminal, "inline" means output embedded in launcher
- **ui.fullscreen**: Whether TUI skill uses fullscreen mode (default: true)
- **context.pass\_\***: Control which environment variables are passed to skill (PANE_CWD, PANE_GIT_ROOT, etc.)

**Context Configuration for claude-tips:**
claude-tips is **self-contained** and does not need project context. All context fields are set to `false`:

- No dependency on current working directory
- No dependency on git repository
- No dependency on project name
- No stdin JSON needed (skill loads tips from bundled data file)

[Source: docs/architecture/data-models.md#contextconfig, docs/prd/epic-4.md#story-4.3-technical-notes]

### Skill Discovery System

**Three-Tier Directory Hierarchy:**
Pane discovers skills by scanning three locations in precedence order:

1. **Project**: `./.pane/skills/**/pane-skill.yaml` (highest precedence)
2. **User**: `~/.config/pane/skills/**/pane-skill.yaml` (medium precedence)
3. **System**: `/usr/local/share/pane/skills/**/pane-skill.yaml` (lowest precedence)

**Discovery Algorithm:**

1. Recursively find all `pane-skill.yaml` files in each location
2. Parse each manifest into a `Skill` struct using `serde_yaml`
3. Validate required fields and executable existence
4. Deduplicate by skill ID using precedence rules (Project > User > System)
5. Return Vec `<Skill>` for launcher display

[Source: docs/architecture/components.md#2-skill-loader, docs/architecture/high-level-architecture.md#2-plugin-discovery-pattern]

**Bundled Skill Installation:**
For MVP, claude-tips will be installed to system skills location:

- Binary: `/usr/local/bin/claude-tips` (in PATH)
- Manifest: `/usr/local/share/pane/skills/claude-tips/pane-skill.yaml`
- Data: `/usr/local/share/pane/skills/claude-tips/data/claude-tips.yaml`

**Alternative**: Skill manifest and data could be embedded in binary using `include_str!` macro for simpler distribution (consider in Story 4.4 or future).

[Source: docs/prd/epic-4.md#story-4.3-technical-notes, docs/architecture/tech-stack.md#distribution]

### Skill Execution Flow

**Primary User Flow (from High Level Architecture):**

1. User runs `pane` command
2. TUI launcher appears with skill search/list interface
3. User selects claude-tips skill (keyboard navigation or search)
4. Pane suspends its TUI and spawns the skill executable
5. claude-tips takes over terminal with its own TUI
6. User interacts with claude-tips (browse, search, view tips)
7. User presses Esc to exit claude-tips
8. claude-tips exits cleanly (exit code 0)
9. Pane restores launcher interface

[Source: docs/architecture/high-level-architecture.md#primary-user-flow]

**Skill Runner Process Management:**
The Skill Runner component (`src/skills/runner.rs`) handles execution:

```rust
fn execute_skill(skill: &Skill, context: SkillContext) -> Result<ExitStatus> {
    // 1. Validate executable exists
    // 2. Prepare environment variables (PANE_* vars)
    // 3. Suspend Pane TUI (disable raw mode, leave alternate screen)
    // 4. Spawn child process with Command::new(skill.exec)
    // 5. Wait for child to exit
    // 6. Restore Pane TUI (enable raw mode, enter alternate screen)
    // 7. Return exit status
}
```

**Terminal Handoff Pattern:**

- **Suspend TUI**: Call `crossterm::terminal::disable_raw_mode()` and `crossterm::execute!(stdout(), LeaveAlternateScreen)`
- **Spawn Child**: Use `std::process::Command::new(&skill.exec).spawn()?.wait()?`
- **Restore TUI**: Call `crossterm::terminal::enable_raw_mode()` and `crossterm::execute!(stdout(), EnterAlternateScreen)`
- **CRITICAL**: Use RAII guard pattern to ensure restoration even on panic

[Source: docs/architecture/components.md#3-skill-runner, docs/architecture/high-level-architecture.md#3-process-management-pattern]

**Context Passing (Not Needed for claude-tips):**
Skills receive context via environment variables:

- `PANE_CWD`: Current working directory
- `PANE_GIT_ROOT`: Git repository root (if detected)
- `PANE_PROJECT_NAME`: Project/repo name
- `PANE_SKILL_ID`: ID of the skill being executed
- `PANE_SKILL_NAME`: Name of the skill

claude-tips does not use any of these (context config all false).

[Source: docs/architecture/data-models.md#skillcontext, docs/architecture/high-level-architecture.md#5-skill-execution-safety-pattern]

### Error Handling

**Critical Rules (MANDATORY):**

1. Never use `unwrap()` or `expect()` in application code
2. Always use `anyhow::Result<T>` for fallible functions
3. Add context to errors using `.context()` or `.with_context()`

**Error Scenarios for Story 4.3:**

- Manifest file not found → "Skill manifest not found at {path}"
- Invalid YAML syntax → "Failed to parse skill manifest: {yaml_error}"
- Missing required field → "Skill manifest missing required field '{field}'"
- Executable not found → "Skill executable '{exec}' not found in PATH or at specified path"

**Validation Requirements:**

- All required manifest fields must be present
- Executable must exist in PATH or be valid absolute path
- UI mode must be "tui" or "inline"
- Tags must be non-empty array
